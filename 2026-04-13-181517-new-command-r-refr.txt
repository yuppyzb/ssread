⏺ 직관은 맞는데, bash 환경에서 "전체 두-풀 분리"는 비용 대비 효과가 애매해요.
  솔직한 평가를 드릴게요.

  왜 두-풀이 끌리는지 (공감)

  실제 상태 축이 이미 많이 엉켜있어요:
  - SESSION_* 15개 병렬 배열 (jsonl 소스)
  - ACTIVE_WINDOWS_STR (tmux 상태)
  - BOOKMARKS_STR, ARCHIVES_STR, SEEN_WORKING_STR (파이프 구분 문자열)
  - ATTACHED_SID (단일)

  각각 소스가 다르고 갱신 주기가 다른데 하나의 sid key로 묶어 lookup 패턴.
  Pending이 추가되면 sid가 없는 상태가 생겨서 이 모델이 깨집니다.

  하지만 bash에서 "두-풀"의 실제 비용

  문제 1: 진짜로 두 풀이 배타적이지 않음
  - Running 세션 = tmux 창 있음 + jsonl 있음 (대부분 경우)
  - Stopped 세션 = jsonl만 있음
  - Pending 세션 = tmux 창만 있음

  한 세션이 동시에 양쪽에 존재하거나 (running) 상태 간 이동(stopped → opened →
  running → closed → stopped)이 잦아요. Bash에 object가 없으니 두 세트의 병렬 
  배열이 되고, 동기화 로직이 또 필요합니다. 결국 15개 × 2 = 30개 배열 + merge
  함수.

  문제 2: 디스플레이는 다시 합쳐야 함
  리스트 한 줄에 다 뿌리는 이상, render 시점에 merge/sort. 복잡도를 줄인 게
  아니라 위치만 바꾼 셈.

  문제 3: 소스 of truth가 진짜로 다른 건 두 종류뿐
  - 디스크(jsonl) — find
  - 런타임(tmux) — tmux list-windows

  나머지(attach/bookmark/seen-working)는 어차피 ssread가 직접 소유하는 in-memory
   플래그. 이걸 "풀"로 분리할 이유는 없음.

  제가 추천하는 중간 지점

  A. 최소 변경 (지금 문제만 해결)
  - PENDING_* 소배열 하나 추가 (sid 없는 애들 전용)
  - SESSION_*는 그대로. 모든 sid-keyed 로직 안 건드림
  - reconcile 시 pending → SESSION_*로 승격
  - 단점: 여전히 "상태 플래그 산재" 구조는 그대로

  B. 라이트 리팩터 (구조 개선 + pending 흡수)
  - SESSION_STATE[i] 배열 1개 추가: stopped | running | pending
  - load_sessions 후 reconcile_sessions() 한 번 돌려서:
    - jsonl 있음 + tmux 창 있음 → running
    - jsonl 있음 + 창 없음 → stopped
    - jsonl 없음 + 창 있음 → pending (sid 자리에 pending:<win_name>)
    - jsonl 없음 + 창 없음 → drop
  - 기존 ACTIVE_WINDOWS_STR lookup → SESSION_STATE[i] 직접 참조로 대체 (lookup
  한 겹 제거)
  - Display / attach / detach 로직이 모두 SESSION_STATE[i] 분기 하나로 단순화
  - Pending도 자연스럽게 리스트에 포함됨 (별도 데이터 경로 없음)

  C. 풀 완전 분리 (당신 제안)
  - 앞서 말한 비용. 굳이 하려면 associative array (declare -A) 기반으로 가야
  깔끔하지만, 그건 거의 재작성.

  제 의견

  B를 추천합니다.

  이유:
  - "실행중 vs 아님"을 명시적으로 분리하고 싶은 본질적 욕구는 상태 필드 하나로
  충족됨
  - sid key는 유지 (pending은 pending:<win_name> 가짜 sid로 통일) → 기존 lookup
  안 깨짐
  - 지금 숨은 버그(detach/재진입)가 SESSION_STATE[i] 분기로 자연스럽게 풀림
  - 추가되는 코드: 배열 1개 + reconcile 함수 1개 (~30줄). 기존 로직 일부
  단순화로 상쇄
  - 테스트 surface가 작음: reconcile 함수만 제대로 돌면 나머지는 기존 경로
  재사용

  C는 "깔끔한 모델"을 얻지만 bash에서 그 모델을 표현하려면 거의 다시 쓰는 비용이

  │ SEEN_WORKING_STR     │ 한 번이라도 작업한 적     │ 내부 플래그          │
  │                      │ 있음 (sticky)             │                      │
  ├──────────────────────┼───────────────────────────┼──────────────────────┤
  │ is_session_working() │ WORKING에 있음            │ 파생                 │
  ├──────────────────────┼───────────────────────────┼──────────────────────┤
  │ is_session_done()    │ SEEN_WORKING && !WORKING  │ 파생                 │
  │                      │ (알림 대상)               │                      │
  └──────────────────────┴───────────────────────────┴──────────────────────┘

  렌더 코드가 세션마다 이 3개 함수를 호출해서 매번 substring 검색. 사실상 이미
  상태 머신인데 enum으로 표현돼 있지 않을 뿐이에요.

  제안: lifecycle vs. flag 분리

  상태를 한 바구니에 넣되, 차원이 다른 것까지 합치면 안 됩니다. 두 축을 구분:

  Lifecycle state (단일값, enum) → SESSION_STATE[i]
  stopped   — jsonl만 있음 (창 없음)
  pending   — 창만 있음 (jsonl 아직 없음)
  idle      — claude 살아있음, 작업한 적 없음
  working   — claude가 tool 실행 중  ← "실행 중"
  done      — 작업했었고 지금은 idle  ← "완료 알림 대상"
  closed    — 창은 있지만 claude 종료됨 (placeholder)

  Orthogonal flags (다중 보유 가능) → 그대로 분리 유지
  - BOOKMARKS_STR, ARCHIVES_STR — 사용자 태그
  - SEEN_WORKING_STR — 내부 sticky 플래그 (SESSION_STATE 계산 재료)
  - ATTACHED_SID — 싱글톤 포인터

  이유: 북마크 + done 은 동시에 참일 수 있어요. 한 세션이 "북마크된 상태로 방금
  완료된" 상황. 이걸 하나의 enum에 쑤셔넣으면 bookmarked-done 같은 조합 폭발이
  발생. Lifecycle은 배타적 단일값, flag는 다중 보유. 성질이 다르니 같은 그릇에
  두면 안 됨.

  장점 (왜 이게 실제로 단순화인지)

  1. 렌더 경로 단순화: 세션당 3번 substring lookup → case ${SESSION_STATE[i]} 한
   방. 성능도 개선.
  2. 전이 지점이 1곳: reconcile_sessions() 하나가 jsonl/tmux/ps/SEEN_WORKING을
  읽어 전체 SESSION_STATE[i]를 다시 계산. 나머지 코드는 읽기만.
  3. 알림(done) 상태가 1급: 지금은 SEEN_WORKING && !WORKING 이라는 파생
  조건인데, done이라는 이름이 붙으면 로직/UI에서 명시적으로 다룰 수 있음 (예:
  알림 카운트, 정렬 우선순위, 소리 훅 등).
  4. Pending이 자연스럽게 편입: 앞서 말한 "new 세션 목록 문제"가 pending이라는
  정식 상태로 해결됨. 특수 케이스 취급 아님.
  5. 숨은 버그 수정이 자연스러워짐: detach/재진입 버그가 SESSION_STATE[i] == 
  pending 분기로 풀림. Ad-hoc prefix 파싱 안 써도 됨.

  주의할 점

  Stateless 욕구와의 타협: 사실 SESSION_STATE[i]는 파생값이에요. 매 refresh 때
  jsonl/tmux/ps 스냅샷에서 재계산. SEEN_WORKING만 sticky이고 나머지는 다 휘발.
  그래서 "진짜 상태"는 여전히 최소화되고, SESSION_STATE[i]는 캐시된 derived 
  view로 보면 돼요. 이 관점이면 원래 원하셨던 stateless 철학에서 크게 벗어나지
  않습니다. (캐시 무효화 규칙만 명확하면 OK — "reconcile은 refresh cycle당
  1회"로 정해두면 됨.)

  리스크: reconcile 타이밍 버그. 예컨대 사용자가 방금 창을 닫았는데 다음
  reconcile 전까지 UI가 stale. 현재도 ACTIVE_WINDOWS_LAST_REFRESH로 2초 쓰로틀링
   중이라 같은 수준. 새로 생기는 문제 아님.

  결론

  찬성입니다. 실제로 상태를 늘리는 게 아니라 이미 분산된 걸 모으는 리팩터에
  가깝고, 이미 있는 복잡도를 가시화하는 방향이에요.

  구현 순서 추천:
  1. SESSION_STATE[] 배열 + enum 상수 정의
  2. reconcile_sessions() 작성 — 기존 refresh_active_windows 로직을 이쪽으로
  흡수하면서 SESSION_STATE 계산
  3. is_session_working/done/active → [[ "${SESSION_STATE[i]}" == working ]]
  식으로 치환 (or 함수 body만 교체)
  4. Pending 지원 추가 (cmd_new_session 시점에 가짜 sid로 엔트리 생성,
  reconcile에서 처리)
  5. detach/attach 로직이 pending 분기 처리
  6. Status bar에 pending 메시지 + done 카운트

  진행할까요? 아니면 먼저 상태 전이 다이어그램(어떤 이벤트가 어떤 전이를
  일으키는지)을 그려서 합의 본 뒤 들어갈까요? 후자를 권합니다 — reconcile 규칙이
   핵심이라 그림 한 장 있으면 구현 중 판단 비용이 확 줄어요.

  3. Reconcile Rule (truth table)

  J  W  C  T  S  →  STATE
  ────────────────────────────
  0  0  -  -  -  →  drop (리스트에서 제거)
  0  1  1  -  -  →  pending
  0  1  0  -  -  →  orphan → 창 kill + drop  (claude가 jsonl 생성 전 종료)
  1  0  -  -  -  →  stopped
  1  1  0  -  -  →  closed
  1  1  1  1  -  →  working     (부수효과: S := 1)
  1  1  1  0  0  →  idle
  1  1  1  0  1  →  done

  이 표 하나만 있으면 reconcile 로직이 끝. 어떤 상태로부터 왔는지 몰라도 OK —
  pure function of snapshot + S.

  8. Edge Cases

  a. pending 도중 claude가 죽음 (Ctrl-C before first message)

  b. jsonl은 있는데 창이 있고 claude는 죽음 (정상 종료)
  - J=1, W=1, C=0 → closed

  - 사용자가 다시 Enter하면 resume 경로로 idle/working

  c. 같은 cwd에서 여러 pending을 띄움
  - pending마다 고유 win_name 보장 필요 (현재 win_name="new" 고정이라 충돌 가능
  → new-<epoch> 같은 suffix 권장)
  - Promotion 매칭 시 cwd만으로는 부족, epoch 비교 필수

  d. Reconcile 중간에 상태가 바뀜
  - ps 읽는 사이 tool이 끝났다면? 다음 cycle에서 바로잡힘. 2초 지연 수용.
  - SEEN_WORKING은 sticky라 "깜빡 보고 놓침" 시나리오도 보호됨.

  e. done 상태의 세션을 archive
  - 직교 축이므로 문제 없음. Archive 후에도 SESSION_STATE는 done 유지.
  - 보통 archive된 세션은 리스트 숨김 → UI에서 안 보이지만 상태는 정확.
